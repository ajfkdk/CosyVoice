1. 我们确定一下关键字，句区，指的是拆分出来的，发送给tts转换为音频的一句或是好几句的的文本内容所生成出来的那个音频所在整体block音频的开始点和结束点
2. 生成的前端页面不要有emoji元素
3. 本项目的关键是把一本小说转换为一个图文形式的动态漫画
4. 项目有一个GUI，有一个新建项目就可以开始创建项目
5. 一个项目相当于一个小说，为这个项目取个名字
6. 进入项目内部点击创建章节就进入这个章节的创建页面
7. 在创建页面中把这个章节的内容放入文本框中（半个屏幕），然后在文本框下面点击开始检测。
8. 开始检测会调用小说句区切分Agent，名字叫做刘备的Agent，刘备会把这段章节拆分为一个个合适的句区
9. 另外半个屏幕会把一个个句区显示出来，可能句区会比较多长，渲染句区的区域是可以滚动的，方便用户查看每个句区裁剪的状态
10. 我要如何裁剪出一个合适的句区呢？句区应该是怎么样的呢？答：这个句区最好是在一个场景中 判断"同一场景"的依据是通过LLM语义理解来判断场景切换（如地点变化、时间跳跃、视角转换  ，如果文字内容大于一定数量比如300字，就自动裁剪出一个新的句区。
11. 双击句区就可以进入句区编辑页面，页面的顶部是一个句区的文本内容，长条，宽度是整个页面的宽度，然后这个句区显示元素下面有两个帮个页面宽度的小div，一个是音频区域，一个是图片生成区域
12. 音频区域有个生成按钮可以点击，点击了生成按钮之后会调用CosyVoice的模型生成带有情感的说书人语音生成，我们称为说书人，调用说书人生成这个句区的音频，带有显示生成进度，生成完成后音频区域有一个音频播放组件，可以马上就听，看看效果怎么样。
13. 当然我还想引入一个人物资产的概念，我可以在项目页面中进入人物资产界面，这个界面会显示各个角色和角色参考图。可以在这里新建角色，新建角色的时候可以让用户输入自己对角色形象的描述，也可以黏贴小说中这个人物的相关描述，然后调用小明Agent自动读取这个人物的一些关键信息，然后生成NanoBanana专属的prompt,最后调用NanoBanana引擎生成三视图，保存到本地作为这个角色的参考图。保存方式为本地人物资产文件夹下的对应角色文件夹(没有就新建）下，保存prompt和对应的角色参考图png
14. 图片生成区域有一个生成prompt按钮，点击之后会调用小乔Agent来生成NanoBanana专属的prompt。在prompt显示文本框下面有一个加号，点击加号可以新增参考图，会弹窗出人物资产modal，这个modal会把人物的图片中等大小显示，图片底部是这个人物的名字，然后一个个人物都遍历显示（瀑布流），选中其中几个人物后这几个人物会作为主角参考图，并自动生成关于参考图人物的描述，自动补充到小乔生成的prompt中，让nanobanana知道其中关键人物是怎么样的，实现人物一致性
15. 点击生图后，进入loading状态，等loading完成后，点击预览就可以查看生成出来的图片。
16. 先用web作为原型设计
17. 我们让nanobanana生成场景图片的时候我们是会给参考图的，而不是把生成参考图的prompt放进去，Nanobanana允许上传参考图图片的，我们只需要在小乔的prompt说明这张参考图分别是谁就可以，因为只支持单图上传所以目前挑选最重要的主角参考图就可以

import requests
import json
import re

API_URL  = "https://api.mmw.ink/v1/chat/completions"

INPUT    = "input.png"              # ← 输入图片路径

# 1. 读取图片 → base64
with open(INPUT, "rb") as f:
    img_b64 = base64.b64encode(f.read()).decode("utf-8")

# 2. 构造请求
payload = {
    "model": "[A]gemini-3-pro-image-preview",
    "messages": [
        {
            "role": "user",
            "content": [
                {
                    "type": "text",
                    "text": PROMPT
                },
                {
                    "type": "image_url",
                    "image_url": {
                        "url": f"data:image/png;base64,{img_b64}"
                    }
                }
            ]
        }
    ]
}

headers = {
    "Authorization": f"Bearer {API_KEY}",
    "Content-Type": "application/json"
}

# 3. 发送请求
print("⏳ 请求中...")
resp = requests.post(API_URL, headers=headers, json=payload, timeout=120)
resp.raise_for_status()
result = resp.json()

# 4. 解析返回的图片
content = result["choices"][0]["message"]["content"]

# 情况 A：返回结构中直接有 base64 图片字段
# 情况 B：返回 markdown 格式 ![](data:image/png;base64,xxxx)
# 情况 C：返回纯 base64 字符串

# 尝试从 markdown 中提取
match = re.search(r"data:image/\w+;base64,([A-Za-z0-9+/=]+)", content)
if match:
    img_data = base64.b64decode(match.group(1))
    with open(OUTPUT, "wb") as f:
        f.write(img_data)
    print(f"✅ 图片已保存到 {OUTPUT}")
else:
    # 如果不是图片，打印文本内容
    print("📝 返回内容（非图片）：")
    print(content[:500])

21. 句区切分Agent"刘备"的容错与人工校正机制是怎样的？
LLM的场景切分不可能100%准确，当刘备切分出的句区不合理时（比如把一个完整对话拆断了，或者两个明显不同的场景合并了），用户除了双击进入编辑页面外，是否需要在句区列表层面支持合并相邻句区、手动拆分单个句区、拖拽调整句区顺序等操作？  回答：需要
22. 最终的"动态漫画"成品输出形态是什么？
你详细描述了句区的音频生成和图片生成流程，但没有提到最终产物。每个句区生成了一张图片和一段音频之后，最终是导出为一个视频（图片配合音频按时间线播放）、还是一个可交互的阅读器（类似条漫翻页+自动播放语音）、还是其他形式？这个问题决定了是否还需要一个"合成/导出"模块，以及是否需要转场动画、字幕叠加、Ken Burns等图片运动效果   回答：先完成一个可交互的阅读器
23. CosyVoice的说书人音色在整个项目中如何统一管理？
从你给的代码看，CosyVoice需要一个参考音频（ref_audio）来克隆音色。这个参考音频是项目级别的配置（整本小说用同一个说书人声音），还是章节级别可以切换？另外，小说中不同角色的对话部分，是否全部用同一个说书人音色去读，还是未来考虑为不同角色分配不同音色（多角色演绎）？这会影响音频引擎的抽象方式和数据库中音频配置的表结构设计。  回答:支持切换说书人参考音频，上传即可，在角色资产中有一个固定的资产是说书人，点击可以上传参考音频
24. NanoBanana目前只支持单张参考图上传，当一个句区涉及多个角色时如何处理？
你提到"目前挑选最重要的主角参考图就可以"，但实际场景中很多句区会同时出现2-3个角色。如果只上传一个角色的参考图，其他角色的外貌一致性如何保证？是否考虑过：将多个角色参考图拼接为一张图再上传、在prompt中用文字详细描述其他角色、或者分多次生成再合成？这个问题对图片质量和角色一致性影响很大，需要提前确定策略。  回答：可以将多个角色参考图拼接为一张图，然后适当降低分辨率，减小大图体积，拼接的时候支持不同角色参考图右边都有角色名字信息。
25. SQLite + FastAPI + Electron的架构中，CosyVoice模型的运行位置和资源管理如何规划？
